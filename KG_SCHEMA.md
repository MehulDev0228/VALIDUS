# KG_SCHEMA.md

This document defines the schema and design for the FutureValidate knowledge graph (KG).

## Data model

Each KG entry represents a startup-relevant artifact (company, pitch, article, etc.).

```ts
interface KGItem {
  id: string
  name: string
  type: "startup" | "pitch" | "article" | "deck"
  text: string              // main body text used for embeddings & similarity
  url?: string              // canonical reference URL
  tags: string[]            // normalized tags like "fintech", "saas", "b2b", "failed", etc.
  createdAt?: string        // ISO timestamp when this artifact was created or scraped
}
```

In the current implementation, this structure is used in:

- `lib/kg/seed.json` – small hand-crafted seed dataset.
- `lib/kg/index.ts` – heuristic token-overlap search on the KG.

## Storage layout

MVP storage is file-based and checked into the repo for reproducibility:

- `lib/kg/seed.json` – base seed entries shipped with the app.
- `data/kg/entries.jsonl` – (optional) additional scraped entries, one JSON object per line matching `KGItem`.
- `data/kg/vector-index.json` – vector index generated by the ingestion pipeline, schema:

```ts
interface KGVectorIndexItem {
  id: string          // KGItem.id
  embedding: number[] // L2-normalized embedding vector
  tags: string[]      // copied from KGItem.tags
}

interface KGVectorIndex {
  model: string              // e.g. "text-embedding-3-small"
  createdAt: string          // ISO timestamp
  items: KGVectorIndexItem[]
}
```

The production app can choose between:

1. **Heuristic search** (current default) – fast and requires only `seed.json`.
2. **Vector search** (future) – uses `vector-index.json` for nearest-neighbour queries.

## Ingestion pipeline (MVP)

The ingestion pipeline is implemented as a Node/TypeScript script `scripts/ingest.ts` and is responsible for:

1. Reading raw KG sources:
   - `lib/kg/seed.json`
   - `data/kg/entries.jsonl` (if present)
2. Optionally fetching external artifacts via scrapers (e.g. Serper.dev searches, public pitch deck libraries).
3. Cleaning and normalizing text & tags.
4. Generating embeddings via OpenAI or a local model.
5. Writing `data/kg/vector-index.json`.

The script is **idempotent** and can be run as a daily cron (e.g. GitHub Actions, Vercel cron, or external scheduler).

## Query strategy

The app currently uses a lightweight heuristic search defined in `lib/kg/index.ts`:

- Tokenize idea input (title, description, industry, targetMarket, revenueModel, keyFeatures).
- Tokenize each `KGItem.text` and union with `tags`.
- Score items by overlap / sqrt(|tokens|).

When vector search is enabled, the flow will be:

1. Embed idea text (same concatenation as above) into a vector.
2. Compute cosine similarity against `KGVectorIndex.items`.
3. Fetch the top-K `KGItem` documents by ID.

## Extending the schema

You can safely extend `KGItem` with additional optional fields, such as:

```ts
interface KGItem {
  // ...existing fields
  stage?: "idea" | "seed" | "series-a" | "growth" | "public"
  outcome?: "unicorn" | "failed" | "acquired" | "indie" | "unknown"
  source?: string  // e.g. "yc", "ph", "hn", "crunchbase", "deck-library"
}
```

When extending, keep the JSONL representation backward compatible (only add optional fields).

## Operational notes

- Embeddings are expensive; prefer batched ingestion runs over per-request embedding.
- When running in production, you should:
  - Move JSONL + vector index to a persistent store (e.g. S3, Supabase storage) if the dataset grows large.
  - Consider moving to a managed vector DB (e.g. Pinecone, Weaviate) if you outgrow local JSON.
- For the MVP, the heuristic search over `seed.json` is sufficient to power the free validation UX.
